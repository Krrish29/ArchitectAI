# ArchitectAI

**ArchitectAI** is an AI-powered Software Architect that transforms a single software idea into a complete, structured technical blueprint — using a coordinated team of specialized AI agents instead of one general-purpose response.

Describe an idea like *"build a Netflix clone"* or *"football quiz app"*, and ArchitectAI's multi-agent system collaboratively produces:

- Functional and non-functional requirements
- System architecture and technology stack
- Database schema (entities, relationships, indexes)
- A complete REST API specification
- A phased implementation roadmap

All five sections are generated by dedicated agents, validated against strict schemas, and rendered live in a React dashboard as each agent completes.

---

## Features

### Core capabilities
- **Multi-agent AI workflow** — Supervisor, Requirement, Architecture, Database, API, and Planner agents, each with a focused responsibility
- **Intelligent Supervisor Agent** — analyzes the idea and selects only the agents relevant to it, rather than always running the full pipeline
- **Dependency-aware execution** — Requirement always runs first; downstream agents (Architecture, Database, API, Planner) receive its output as context
- **Editable blueprint sections** — requirements can be edited and saved back into the project after generation
- **Project history** — past blueprints are listed and revisitable from the sidebar
- **Local persistence** — projects are saved client-side, no account required

### Reliability
- **Schema-validated agent output** — every agent response is checked against a strict Pydantic schema before reaching the frontend, with automatic coercion for common shape mismatches (e.g. a list of strings where an object was expected)
- **Graceful degradation** — if one agent fails (a malformed or incomplete LLM response), the rest of the pipeline still completes; the failure is isolated and logged rather than crashing the whole request
- **No silent data loss** — defaults and fallbacks are explicit, so a missing field surfaces as a visible gap rather than disappearing without explanation

### Interface
- Modern, responsive dashboard built with React and Tailwind CSS
- Clean dark theme with consistent iconography across all blueprint sections
- Per-section copy-to-clipboard for pulling content into other tools

## Tech stack

| Layer | Technologies |
|---|---|
| **Frontend** | React, Vite, Tailwind CSS, Context API, LocalStorage, Lucide React, Axios |
| **Backend** | FastAPI, Pydantic, Python |
| **LLM runtime** | Ollama, running Qwen2.5:3B locally |

Running the LLM locally via Ollama means no API keys, no per-request cost, and no external dependency at inference time — at the tradeoff of a smaller model than hosted frontier alternatives, which is part of why ArchitectAI leans on strict schema validation and explicit prompting rather than assuming well-formed output by default.

## System architecture

```text
User idea
   │
   ▼
FastAPI Backend
   │
   ▼
Supervisor Agent  ──── selects which agents are needed for this idea
   │
   ▼
Requirement Agent  ──── always runs; output feeds every downstream agent
   │
   ├──────────────┬──────────────┬──────────────┐
   ▼              ▼              ▼              ▼
Architecture    Database        API          Planner
  Agent          Agent         Agent          Agent
   │              │              │              │
   └──────────────┴──────────────┴──────────────┘
                       │
                       ▼
              Final Blueprint Response
                       │
                       ▼
              React Dashboard
```

Architecture, Database, API, and Planner each run independently once Requirement completes, conditioned on what the Supervisor Agent decided is relevant to the idea.

## AI agents

### Supervisor Agent
Analyzes the user's idea and selects which downstream agents are actually needed. Falls back to running all agents if its own output can't be parsed, so a single bad LLM response never blocks generation entirely.

### Requirement Agent
Generates:
- Project name
- Features
- Functional requirements
- Non-functional requirements

### Architecture Agent
Generates:
- Architecture type (e.g. microservices, monolith)
- Frontend, backend, database, and deployment technology choices
- Additional architectural features with implementation detail

### Database Agent
Generates:
- Database engine selection
- Entities, each with type and description
- Relationships between entities
- Index recommendations

### API Agent
Generates:
- REST endpoints covering the generated database entities
- Request schemas
- Response schemas

### Planner Agent
Generates:
- Development phases
- Tasks per phase
- Milestones and estimated timeline

## How it works

1. The user describes a software idea.
2. The Supervisor Agent analyzes the idea and selects the relevant downstream agents.
3. The Requirement Agent always runs first, producing the base context every other agent uses.
4. Selected agents (Architecture, Database, API, Planner) execute, each receiving the Requirement output as input.
5. Each agent's raw LLM response is parsed, validated against its schema, and normalized — malformed or incomplete output is coerced into a safe default rather than breaking the pipeline.
6. The complete blueprint is assembled and returned to the frontend.
7. The React dashboard renders each section as soon as the full response arrives, with the agent timeline showing which agents ran and completed.

## Installation

### Clone the repository

```bash
git clone https://github.com/your-username/ArchitectAI.git
cd ArchitectAI
```

### Backend setup

```bash
cd backend

python -m venv venv

# Windows
venv\Scripts\activate

# macOS / Linux
source venv/bin/activate

pip install -r requirements.txt
```

Start Ollama:

```bash
ollama serve
```

Pull the model:

```bash
ollama pull qwen2.5:3b
```

Run the backend:

```bash
uvicorn app.main:app --reload
```

### Frontend setup

```bash
cd frontend
npm install
npm run dev
```
